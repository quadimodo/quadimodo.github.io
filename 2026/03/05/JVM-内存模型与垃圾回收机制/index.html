二、面试高频题

Q1: JVM 有哪些内存区域？各有什么作用？

A1: 推荐回答

JVM 运行时数据区分为线程共享和线程私有两大类：

线程共享区域：

  1. 方法区：存储类信息、常量、静态变量。JDK 8 后用元空间（Metaspace）实现，使用本地内存。
  2. 堆：存储对象实例，是垃圾收集的主要区域。分为新生代和老年代。

线程私有区域：

  1. 虚拟机栈：每个方法调用创建一个栈帧，存储局部变量表、操作数栈等。方法调用就是栈帧入栈出栈。
  2. 本地方法栈：为 Native 方法服务。
  3. 程序计数器：记录当前线程执行的字节码行号，保证线程切换后能恢复执行。

常见异常：

  • 堆溢出：OutOfMemoryError: Java heap space
  • 栈溢出：StackOverflowError（递归过深）或 OutOfMemoryError（创建太多线程）
  • 方法区溢出：OutOfMemoryError: Metaspace（加载太多类）

Q2: 对象在堆中是如何分配的？

A2: 推荐回答

对象在堆中的分配遵循以下规则：

1. 优先在 Eden 区分配

  • 大多数对象生命周期很短
  • Eden 区满后触发 Minor GC

2. 大对象直接进老年代

  • 避免在 Survivor 区大量复制
  • 通过 -XX:PretenureSizeThreshold 设置阈值

3. 长期存活对象进老年代

  • 对象在 Survivor 区每熬过一次 GC，年龄 +1
  • 默认 15 岁晋升老年代（-XX:MaxTenuringThreshold）

4. 动态年龄判定

  • Survivor 区相同年龄对象总大小 > 空间的一半
  • 年龄 ≥ 该年龄的对象直接进老年代

5. 空间分配担保

  • Minor GC 前检查老年代可用空间
  • 如果不足，可能触发 Full GC

分代原因：不同对象生命周期不同，分代可以用不同算法优化 GC 性能。新生代对象存活率低，用复制算法；老年代对象存活率高，用标记-整理算法。

Q3: 如何判断对象可以被回收？

A3: 推荐回答

JVM 使用可达性分析法判断对象是否可回收：

原理：
从 GC Roots 开始向下搜索，搜索走过的路径称为引用链。如果一个对象到 GC Roots 没有任何引用链相连，则该对象不可达，可以被回收。

GC Roots 包括：

  1. 虚拟机栈中引用的对象（局部变量）
  2. 方法区中类静态属性引用的对象
  3. 方法区中常量引用的对象
  4. 本地方法栈中 JNI 引用的对象
  5. 同步锁持有的对象

为什么不使用引用计数法？

  • 无法解决循环引用问题
  • 例如：A 引用 B，B 引用 A，都不为 0，无法回收

对象真正死亡过程：

  1. 第一次标记：发现不可达
  2. 筛选：判断是否有必要执行 finalize() 方法
  3. 第二次标记：finalize() 中如果重新建立引用，则复活；否则回收

注意：不建议使用 finalize()，已废弃。

Q4: 有哪些垃圾回收算法？各有什么优缺点？

A4: 推荐回答

1. 标记-清除（Mark-Sweep）

  • 优点：简单，不需要移动对象
  • 缺点：
    • 效率低（需要两次遍历）
    • 内存碎片（不连续）
    • 大对象难以分配
  • 适用：老年代（CMS 使用）

2. 复制算法（Copying）

  • 优点：
    • 简单高效
    • 无内存碎片
    • 适合对象存活率低的场景
  • 缺点：内存利用率低（需要 2 倍空间）
  • 优化：Eden : S0 : S1 = 8 : 1 : 1，每次只浪费 10%
  • 适用：新生代（Serial、ParNew、Parallel Scavenge）

3. 标记-整理（Mark-Compact）

  • 优点：
    • 无内存碎片
    • 适合对象存活率高的场景
  • 缺点：效率低（需要移动对象并更新引用）
  • 适用：老年代（Serial Old、Parallel Old）

4. 分代收集算法

  • 根据对象存活周期分代
  • 新生代：复制算法（对象存活率低）
  • 老年代：标记-清除 或 标记-整理

总结：没有完美算法，需要根据场景选择。现代 JVM 使用分代收集，不同代用不同算法。

Q5: CMS 和 G1 有什么区别？

A5: 推荐回答

CMS（Concurrent Mark Sweep）：

  • 目标：低延迟
  • 算法：标记-清除
  • 流程：
    1. 初始标记（STW）：标记 GC Roots 直接关联的对象
    2. 并发标记：从 GC Roots 遍历整个对象图
    3. 重新标记（STW）：修正并发标记期间的变动
    4. 并发清除：清除垃圾对象
  • 优点：停顿时间短，用户体验好
  • 缺点：
    • 内存碎片（标记-清除）
    • CPU 敏感（并发占用资源）
    • 浮动垃圾（并发清除时新产生的垃圾）
  • 适用：互联网应用、低延迟要求

G1（Garbage First）：

  • 目标：可预测停顿时间
  • 算法：复制 + 标记-整理（Region 级别）
  • 特点：
    • 将堆分为多个 Region（2048 个）
    • 优先回收垃圾最多的 Region（Garbage First）
    • 可以设置期望停顿时间
  • 优点：
    • 无内存碎片
    • 可预测停顿
    • 适合大堆（6GB+）
  • 缺点：内存占用高（每个 Region 需要记忆集）
  • 适用：大堆、低延迟、可预测停顿

主要区别：

特性 CMS G1
算法 标记-清除 复制 + 整理
内存碎片 有 无
停顿时间 不可预测 可预测
堆大小 中小堆 大堆（6GB+）
JDK 版本 1.5+ 1.7+

选择建议：

  • 堆 < 6GB，低延迟：CMS
  • 堆 ≥ 6GB，可预测停顿：G1
  • 极低延迟（< 10ms）：ZGC

Q6: 什么时候会触发 Full GC？

A6: 推荐回答

触发 Full GC 的场景：

1. 显式调用

2. 老年代空间不足

  • 大对象直接进老年代，老年代满
  • 长期存活对象晋升，老年代满
  • Minor GC 前担保检查失败

3. 方法区/元空间满

  • 加载太多类
  • 常量池满

4. CMS 失败

  • 并发模式失败（老年代在并发清除时满）
  • 晋升失败（ Survivor 区放不下，老年代也放不下）

5. G1 的 Allocation Failure

  • 没有空闲 Region 可分配
  • 触发 Full GC 进行整理

6. jmap -histo:live

  • 触发 Full GC 后打印对象统计

避免 Full GC 的方法：

  1. 调大新生代，减少晋升
  2. 降低进入老年代的年龄阈值
  3. 使用 G1，避免 Full GC
  4. 优化代码，减少大对象
  5. 避免显式调用 System.gc()

Full GC 影响：

  • 停顿时间长（可能几秒到几十秒）
  • 影响用户体验
  • 生产环境应尽量避免

四、延伸思考

4.1 为什么不用引用计数法？

原因：

  1. 循环引用无法回收
  1. 效率问题
  • 每次引用变化都要更新计数器
  • 频繁的写操作影响性能
  1. 空间开销
  • 每个对象都要存储引用计数器

Python 使用引用计数法：

  • 配合标记-清除解决循环引用
  • 但 JVM 选择可达性分析

4.2 为什么分代？

原因：

  1. 对象生命周期不同
  • 80%-90% 对象生命周期很短
  • 少数对象长期存活
  1. 优化 GC 性能
  • 新生代：对象存活率低 → 复制算法（只复制少量存活对象）
  • 老年代：对象存活率高 → 标记-整理（避免复制大量对象）
  1. 减少 STW 时间
  • Minor GC 只回收新生代
  • 老年代 GC 频率低

如果用相同算法：

  • 每次都要扫描整个堆
  • STW 时间长
  • 性能差

4.3 CMS 为什么被废弃？

JDK 9 标记废弃，JDK 14 移除

原因：

  1. 内存碎片
  • 标记-清除算法
  • 大对象难以分配
  1. 浮动垃圾
  • 并发清除时产生新垃圾
  • 需要预留空间
  1. CPU 敏感
  • 并发线程占用 CPU
  • 默认线程数 = (CPU 核数 + 3) / 4
  1. 失败后回退
  • 并发模式失败 → Serial Old（单线程）
  • 停顿时间很长
  1. G1 更优秀
  • 无碎片
  • 可预测停顿
  • 适合现代硬件
